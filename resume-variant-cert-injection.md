# Resume Variants — Append-Only Cert Injection

**Status: Approach approved 2026-07-07 (append-only basis) — implementation not yet started.**

Supersedes `resume-restructure-typed-data-module.md` (the full typed-data-module rewrite), which is **withdrawn**.

## Decision history (why append-only, not the rewrite)

A review of `develop` vs `main` surfaced nine findings, traced to one root cause: the variant flow *subtracts* from a fixed markdown body and so cannot *add* content (e.g. certifications) for a specific target. Two fixes were weighed:

- **Full rewrite** — migrate the resume to a typed data module (`src/data/resume.ts`) and assemble the default and every variant from it. Clean single source, but a large migration that routes the *published* default through new, unproven code, moves prose-heavy content out of markdown into TS strings, and makes each variant a full site build.
- **Append-only (chosen)** — keep the published markdown default exactly as-is; add the one missing capability (inject certifications) to the existing variant builder.

The deciding factor: **variants are disposable, one-time knock-offs.** Only the default publishes; every variant is generated locally, eyeballed, sent, and discarded. That makes the rewrite's headline benefit (a robust, source-driven variant pipeline) low-value, and neutralizes append-only's headline cost (fragile DOM-scrape / hand-maintained facet tags only ever touch throwaway output that is visually checked). Under that weighting, append-only carries less operational overhead.

Retained cosmetic debt — inline `<!-- f: … -->` tags in the markdown and `data-facets` in the public HTML — is standards-valid and SEO-inert: custom `data-*` attributes are not a ranking signal, are not indexed as content, are not structured data, and are distinct from crawlable "faceted navigation." Accepted.

## The default-view guarantee

The published default is untouched:

- `src/content/pages/resume/index.md` **body** stays byte-for-byte identical; only inert frontmatter is removed.
- None of the removed frontmatter is referenced by either route (`resume.astro`, `resume/print/index.astro`) — verified: zero usages outside the schema — so the rendered default is **byte-identical**. No visual regression is possible.
- **Certifications have never rendered on the published default:** neither route reads the `certifications` frontmatter — both render `<Content />` (body) plus contact fields only — so moving certs to `certifications.json` removes nothing from any shipped output. A cert section appears *only* in a variant that opts in via `include_certs`.
- The variant builder never runs `astro build` into `dist/` and writes its PDF to an out-of-repo path. It cannot mutate the canonical resume, the public page, or the shipped `kyle-skrinak-resume.pdf`. (This satisfies the earlier "guard against a stray variant build" concern *by construction* — there is no env-driven build to leak.)

## Scope

**Kept, unchanged:** the markdown body and both routes; `src/styles/resume.css` (byte-identical — the cert-heading style is **runtime-injected** by the transform, never added to the shipped stylesheet); `src/lib/remark-facets.mjs` + `data-facets` (they drive variant facet-filtering); the one-page gate logic (`countPdfPages`); CI's default resume-PDF regeneration.

**Modified (backward-compatible):**

Already a resume-PDF hash input (`production-deploy.yml:103`) — a content change, **no new hash-path**.
- `scripts/build-resume-variant.mjs` + its `validateConfig` — cert read/resolve/inject, the injected cert-heading `<style>` (F‑1, `::before` left intact per G‑2), the positional `<ul>` self-check and Node-side fallback warn (G‑4/G‑1), and the `include_certs` rule; refactored for testability (step 5).

**Removed:** the inert Phase-A frontmatter (feature PR); its schema is a **decoupled cleanup PR** (G‑3, see "Deletions" / step 2).

**Added:** a durable `certifications.json` (cert list + `anchor_before_id`), an `include_certs` variant-config field, cert injection in the variant transform, and one variant test.

**Not done** (vs. the withdrawn plan): no `src/data/resume.ts`, no `assembleResume`, no `ResumeBody.astro`, no render-path change, no per-variant full build, no `astro.config.ts` processor revert, no Phase B/C reversion. Append-only is purely additive on top of current `develop`.

## Certification data — `scripts/data/certifications.json`

Durable, versioned, local-tool-only (the site build never reads it). Selection is by explicit id per variant. **Lives in `scripts/data/`, not `src/`** — the variant builder is in `scripts/` and reads the file by absolute path, so it has no reason to sit among the build-*consumed* `src/data/*.json` (`presentations.json` / `socialLinks.json`). Keeping it out of `src/` **is** the safeguard, not a comment: a tool-only tree can't be swept into the bundle by a stray `src/data` glob and can't be caught by a `hash-paths` list scoped under `src/` — so **no `"_comment"` marker and no hash-paths audit are needed** (both retired vs. the earlier `src/data` proposal). `loadCertifications()` validates `certifications` + `anchor_before_id` and ignores any other top-level keys (forward-compat).

```json
{
  "anchor_before_id": "ms-it",
  "certifications": [
    { "id": "aiops-foundation", "name": "AIOps Foundation", "issuer": "PeopleCert", "issued": "2026-01" },
    { "id": "az-104", "name": "Microsoft Certified: Azure Administrator Associate (AZ-104)", "issuer": "Microsoft" }
  ]
}
```

- Cert fields: `id` (stable, explicit), `name` (required), `issuer?`, `issued?` (`YYYY-MM`), `facets?` (optional; unused today, reserved for future facet-based selection).
- `anchor_before_id` (required when any variant injects certs): the rendered `h2` id to inject the Certifications section **before** (e.g. `ms-it`, the M.S. heading — there is no literal "Education" heading; education is per-degree `h2`s). **Placement is deliberate:** injecting before the first education `h2` lands certs immediately above the degree block — the conventional certs-adjacent-to-education position — not an artifact of "nearest convenient anchor." A cert-emphasis variant that wants them higher changes this **one field** (e.g. an employer `h2` id); the anchor stays **file-global** (one per `certifications.json`) under the disposability thesis, not a per-variant knob. Resolved by a query **scoped to the transform's `.resume-content` root** — `content.querySelector('#' + CSS.escape(anchor_before_id))`, not `document.getElementById`, so an id collision elsewhere in the print document can't misdirect the injection. **The end-of-content fallback is a low-probability *live* path, not merely defensive:** the id is pre-validated against the derived entry-id set and the transform never removes headings, so the scoped lookup misses **only** if the accepted slug-equivalence divergence (see Accepted limitations) makes the DOM id differ from the derived id — the exact drift the fallback exists for. Because a miss would relocate the section past education while still passing the one-page gate and the cert-presence checks (the names still render, just in the wrong place), the transform **signals the fallback by return value, not an in-page `console.warn`**: `page.evaluate` returns `{ emptiedEntries, certAnchorFellBack }` and the **Node** side warns — mirroring the emptied-entry path (returned at `build-resume-variant.mjs:214`, warned in Node at `:219`). A `console.warn` *inside* the evaluate would land in the browser console, never the terminal (the code comments on exactly this at `:217‑218`); page-context **exceptions** propagate to Node but **`console` output does not**. So the misplacement surfaces in the terminal, not left to the eyeball alone (G‑1).
- **Shape-validated on load** (infrastructure code): `certifications` is a non-empty array; each entry has a unique, non-empty `id` and non-empty `name`; `anchor_before_id` ∈ the derived entry-id set (from `deriveEntryIds()`, the same set that already validates `bullet_order` keys) — an unknown anchor or malformed file **fails fast** with a clear message before any render. **Gating:** "certs requested" is the predicate `Array.isArray(include_certs) ? include_certs.length > 0 : include_certs === "all"` — an empty array `[]` requests nothing (file optional), the string `"all"` requests certs (file required). The file is read + validated *only* when that predicate is true. When no variant requests certs the file is never touched and may be absent (default/no-cert path unaffected); a requested-but-absent (or malformed) file **is** a fail-fast error (certs asked for, no cert data).
- The old frontmatter `render: true` flag is dropped — per-variant `include_certs` replaces a global on/off (this retires the "unused render flags" finding).

## Variant config — `include_certs`

New optional field on the existing JSON variant config:

```json
{
  "title": "Kyle Skrinak — Platform Operations Leader",
  "include_facets": ["platform-ops", "leadership", "cost"],
  "max_bullets_per_entry": 3,
  "include_certs": ["az-104", "aiops-foundation"]
}
```

- `include_certs`: array of cert ids [chosen], or the literal `"all"`. Two-part validation in `validateConfig`: (1) a **type/shape** check (must be a string array or exactly `"all"`) that runs **independent of** `knownCertIds`, so a malformed value (`123`, an object, a mixed array) is rejected regardless of whether certs were loaded, **and duplicate ids are rejected** (`new Set(include_certs).size !== include_certs.length`) so a repeated id (e.g. `["az-104", "az-104"]`) can't render the same cert twice — a copy-paste slip, caught fail-fast like every other malformed value; (2) an **id-membership** check against the injected `knownCertIds` set — the only part that needs the loaded file. A bad id, duplicate id, or bad shape fails fast with a clear message *before any render*.
- Omitted / empty ⇒ no certifications section (matches the default).

## Cert rendering (variant only)

- The builder reads `certifications.json`, resolves the requested ids (or `"all"`), and passes the resolved certs into the existing `page.evaluate` transform. **Single-arg bundling:** Playwright's `page.evaluate(fn, arg)` accepts exactly **one** argument — the current call passes `config` alone (`build-resume-variant.mjs:137,215`); a naive `page.evaluate((cfg, certs) => …, config, certs)` silently drops `certs` (it arrives `undefined` in page context, injecting nothing). Bundle instead: `page.evaluate(({ cfg, certs }) => …, { cfg: config, certs })`, threading via `buildTransform(config, resolvedCerts)`. If botched (certs arrive `undefined` in page context → nothing injected), the **positional `<ul>` self-check throws in-page** on the absent names (G‑4, below) rather than shipping silently — but bundling correctly avoids the detour. **Order:** explicit `include_certs` renders in **requested order** (first id → first `<li>`); `"all"` renders in file order.
- **Anchor:** the transform locates the anchor `<h2>` via `content.querySelector('#' + CSS.escape(anchor_before_id))` (scoped to `.resume-content`) and injects an `<h2 class="cert-heading">Certifications</h2>` + `<ul>` — real DOM nodes built with `document.createElement`, not injected markdown — immediately before it; if the id is absent it appends at end-of-content and **returns a `certAnchorFellBack` flag so the Node side warns** (low-probability live path, per the `anchor_before_id` note above; the warn is Node-side because page-context `console` output never reaches the terminal, G‑1). The id is validated against the derived entry-id set pre-spawn, so a stale anchor fails before the browser launches.
- **Cert-heading styling (F‑1):** the injected `<h2>` **cannot** blindly reuse the resume's `h2` CSS. In `@media print` — the media the PDF renders — `resume.css` styles every entry `h2` as `display: inline` (`:177`) with `h2::after { content: ", " }` (`:192`), a rule built to run the role into the inline employer `<p>` that follows (`h2 + p { display: inline }`, `:197`). The cert `<h2>` is followed by a `<ul>`, not an employer `<p>`, so those inherited rules would print a dangling "**Certifications,**" (orphan comma, then the list breaks to the next line). Fix **without touching the shipped stylesheet**: the transform injects a scoped `<style>` node — `.resume-content h2.cert-heading { display: block } .resume-content h2.cert-heading::after { content: none }` — keeping `resume.css` byte-identical (it is a resume-PDF hash input **and** the published default's stylesheet) and the override contained to the variant tool. `::after` is a pseudo-element, so it **cannot** be reset via an inline `style` attribute on the node — a rule (class selector) is required, hence the injected `<style>`, not `element.style`. **The sibling `h2::before` block rule (`resume.css:181` — `content:""; display:block; margin-top:0.75rem`, with a `:first-of-type` reset at `:188`) is *deliberately not* overridden (G‑2):** with the cert heading now `display:block`, `::before` supplies the same inter-entry top gap every other entry gets, and the heading's own screen-rule `margin-top:0.75rem` collapses with it rather than doubling — so certs sit above the education block with correct spacing. Only `display` and `::after` need neutralizing; carrying the plan's "can't blindly reuse the entry-`h2` CSS" thesis through to `::before` confirms it is a feature, not a leak. The cert `<ul>`/`<li>` **do** reuse the existing list CSS cleanly (same `.resume-content ul`/`li` selectors, same depth).
- **Injection order:** the cert section is added **after** the existing bullet-walker loop, so the injected `<ul>` is **exempt** from facet-filtering, `max_bullets_per_entry`, and `bullet_order` — certs are all-or-nothing, never silently capped or reordered by a job-bullet knob. (Injecting before the loop would let `max_bullets_per_entry: 3` silently drop the 4th+ cert — explicitly avoided.)
- **List item:** each `<li>` renders the cert `name`, optionally followed by ` — {issuer}` and ` ({issued})` when present. Only `name` is content-verified — and verified **positionally**, asserted inside the injected `<ul>` (G‑4), not merely present somewhere in the body.
- **One-page budget:** certs compete for space like any other content — the existing one-page gate (`countPdfPages`) fails the variant build if the additions overflow, and the fix is to trim **job** bullets (`max_bullets_per_entry` / `exclude_facets`) **or select fewer certs** (`include_certs` — certs are all-or-nothing, so dropping ids is the only cert-side lever), consistent with today's "trim weakest bullets first" rule. Each resolved cert `name` is passed as `requireText` so `verifyRenderedContent` asserts it in the rendered `innerText` before the PDF is written (see Scope → `resume-render.mjs`) — a whole-body cross-source check **secondary** to the positional `<ul>` self-check (G‑4), since a whole-body `includes` can false-pass on a name that appears elsewhere. **Message hint:** the current gate error says only "trim weakest bullets". That message lives in the **shared** `renderResumePdf` core, which the default CI render (`print-resume-pdf.mjs`) also runs — so the cert clause must be **conditional**, appended only when `expectedOverrides?.requireText?.length` (i.e. a cert variant). **Guard the object, not just the field:** `renderResumePdf` destructures `expectedOverrides` with **no default** (`resume-render.mjs:120-126`) and `print-resume-pdf.mjs:43` passes none, so it is `undefined` on the default CI path — a bare `expectedOverrides.requireText?.length` would throw `TypeError: Cannot read properties of undefined` inside the one-page error block (`resume-render.mjs:204`), replacing the helpful message with a crash exactly when the default resume overgrows one page. Use `expectedOverrides?.requireText?.length` (or add `expectedOverrides = {}` to the signature). This differs from the Step-4 spread inside `verifyRenderedContent`, which is safe only because *that* function defaults its param to `{}` (`resume-render.mjs:80`) — the two sites have different null-safety, so the same expression can't be copied between them. This names certs as a possible overflow cause on variant builds (which have no per-cert trim lever — certs are all-or-nothing, so the appended clause points the user to reducing `include_certs`) while keeping the default path's failure text cert-free.

## Accepted limitations (disposability-justified)

- **Positional bullet reorder** (`bullet_order`) stays index-based against the post-filter list — the known foot-gun is retained because variants are visually checked and disposable; not worth adding stable bullet ids to the markdown.
- **`data-facets` in public HTML** and **inline facet tags in the markdown** remain — cosmetic, SEO-inert, accepted.
- **Custom `unified()` processor** in `astro.config.ts` stays (drives `remark-facets`) — one low-churn version-pin watch-item; may be reverted later if `remark-facets` runs cleanly in the plain `remarkPlugins` array (out of scope here).
- **Slug-equivalence dependency** — the cert anchor resolves `anchor_before_id` against DOM ids from Astro's rehype-slug, while pre-validation derives the same ids with standalone `github-slugger`. `bullet_order` already relies on this equivalence; the cert anchor is now a **second** feature that breaks if the two slug implementations ever diverge. One low-probability watch-item, accepted — and **de-fanged** for the cert path: a divergence makes the scoped anchor lookup miss, which now appends at end-of-content and **warns Node-side via a returned `certAnchorFellBack` flag** (G‑1/F‑4) rather than silently misplacing the section.

## Verification / control model

- **Default:** byte-identical render is the proof; `test:visual` stays green with no baseline change. No pixel-perfect migration gate is needed because the default does not change.
- **Variant:** the existing one-page gate + **positional cert-injection self-check** (each `name` asserted inside the injected `<ul>`, G‑4) + content-presence verification (`requireText`, secondary) + a one-time visual eyeball. No pixel baseline for disposable output.
