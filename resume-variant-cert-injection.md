# Resume Variants — Append-Only Cert Injection

**Status: Implemented on `develop` 2026-07-08. Historical implementation plan retained for context.**

Supersedes `resume-restructure-typed-data-module.md` (the full typed-data-module rewrite), which is **withdrawn**.

## Decision history (why append-only, not the rewrite)

A review of `develop` vs `main` surfaced nine findings, traced to one root cause: the variant flow *subtracts* from a fixed markdown body and so cannot *add* content (e.g. certifications) for a specific target. Two fixes were weighed:

- **Full rewrite** — migrate the resume to a typed data module (`src/data/resume.ts`) and assemble the default and every variant from it. Clean single source, but a large migration that routes the *published* default through new, unproven code, moves prose-heavy content out of markdown into TS strings, and makes each variant a full site build.
- **Append-only (chosen)** — keep the published markdown default exactly as-is; add the one missing capability (inject certifications) to the existing variant builder.

The deciding factor: **variants are disposable, one-time knock-offs.** Only the default publishes; every variant is generated locally, eyeballed, sent, and discarded. That makes the rewrite's headline benefit (a robust, source-driven variant pipeline) low-value, and neutralizes append-only's headline cost (fragile DOM-scrape / hand-maintained facet tags only ever touch throwaway output that is visually checked). Under that weighting, append-only carries less operational overhead.

Retained cosmetic debt — inline `<!-- f: … -->` tags in the markdown and `data-facets` in the public HTML — is standards-valid and SEO-inert: custom `data-*` attributes are not a ranking signal, are not indexed as content, are not structured data, and are distinct from crawlable "faceted navigation." Accepted.

## The default-view guarantee

The published default is untouched:

- `src/content/pages/resume/index.md` keeps the published resume prose and rendered structure intact; inert facet comments are present in the markdown body for local variant filtering, and inert frontmatter is removed.
- None of the removed frontmatter is referenced by either route (`resume.astro`, `resume/print/index.astro`) — verified: zero usages outside the schema — so the rendered default remains content-identical. Visual regression testing remains the proof.
- **Certifications have never rendered on the published default:** neither route reads the `certifications` frontmatter — both render `<Content />` (body) plus contact fields only — so moving certs to `certifications.json` removes nothing from any shipped output. A cert section appears *only* in a variant that opts in via `include_certs`.
- The variant builder never runs `astro build` into `dist/`. In the normal bare-name flow its default PDF path sits next to the out-of-repo variant config, so it does not automatically mutate the canonical resume, the public page, or the shipped `kyle-skrinak-resume.pdf`. This is a default/operator-safety guarantee, not path enforcement: an explicit `--output` or an in-repo literal config path can still intentionally write inside the working tree. No new path guard is added.

## Scope

**Kept conceptually unchanged:** the published resume prose, both resume routes, the one-page gate logic (`countPdfPages`), and CI's default resume-PDF regeneration. The markdown body carries inert facet comments for local variant filtering. The cert-heading style is **runtime-injected** by the transform, never added to the shipped stylesheet.

**Modified (backward-compatible):**

Already a resume-PDF hash input (`production-deploy.yml:103`) — a content change, **no new hash-path**.
- `scripts/build-resume-variant.mjs` + its validation helpers — cert read/resolve/inject, the injected cert-heading `<style>` (F‑1, `::before` left intact per G‑2), the positional `<ul>` self-check, hard failure on a missing cert anchor (G‑4/G‑1), and the `include_certs` rule; refactored for testability and import safety (step 5).

**Removed:** the inert Phase-A frontmatter (feature PR); its schema is a **decoupled cleanup PR** (G‑3, see "Deletions" / step 2).

**Added:** a durable `certifications.json` (cert list + `anchor_before_id`), an `include_certs` variant-config field, an optional per-variant `anchor_before_id` override, cert injection in the variant transform, variant unit tests, and a `linkedom`-based DOM injection test.

## Certification data — `scripts/data/certifications.json`

Durable, versioned, local-tool-only (the site build never reads it). Selection is by explicit id per variant. **Lives in `scripts/data/`, not `src/`** — the variant builder is in `scripts/` and reads the file by absolute path, so it has no reason to sit among the build-*consumed* `src/data/*.json` (`presentations.json` / `socialLinks.json`). Keeping it out of `src/` **is** the safeguard, not a comment: a tool-only tree can't be swept into the bundle by a stray `src/data` glob and can't be caught by a `hash-paths` list scoped under `src/` — so **no `"_comment"` marker and no hash-paths audit are needed** (both retired vs. the earlier `src/data` proposal). `loadCertifications()` validates `certifications` + `anchor_before_id` and ignores any other top-level keys (forward-compat).

```json
{
  "anchor_before_id": "ms-it",
  "certifications": [
    { "id": "aiops-foundation", "name": "AIOps Foundation", "issuer": "PeopleCert", "issued": "2026-01" },
    { "id": "az-104", "name": "Microsoft Certified: Azure Administrator Associate (AZ-104)", "issuer": "Microsoft" }
  ]
}
```

- Cert fields: `id` (stable, explicit, non-empty), `name` (required, non-empty), `issuer?` (non-empty when present), `issued?` (`YYYY-MM`). A `facets` field is **not accepted** — the validator rejects it until facet-based cert selection is implemented. Re-add to the schema at that time.
- `anchor_before_id` (required when any variant injects certs): the rendered `h2` id to inject the Certifications section **before** (e.g. `ms-it`, the M.S. heading — there is no literal "Education" heading; education is per-degree `h2`s). **Placement is deliberate:** injecting before the first education `h2` lands certs immediately above the degree block — the conventional certs-adjacent-to-education position — not an artifact of "nearest convenient anchor." The `certifications.json` value is the **default** anchor; a variant config can override it with its own `anchor_before_id` field (see "Variant config" below), so a cert-emphasis variant that wants certs placed higher (e.g. before an employer `h2`) does not require editing the durable file. Resolved by a query **scoped to the transform's `.resume-content` root** — `content.querySelector('#' + CSS.escape(anchor_before_id))`, not `document.getElementById`, so an id collision elsewhere in the print document can't misdirect the injection. There is **no end-of-content fallback by default**: a miss means stale anchor data, slug-equivalence drift, or unexpected DOM shape, and the transform throws in page context before any PDF is written. If a fallback is ever useful, make it an explicit opt-in variant/tool flag, not the default.
- **Shape-validated on load** (infrastructure code): `certifications` is a non-empty array; each entry has a unique, non-empty `id` and non-empty `name`; optional `issuer` is a non-empty string; optional `issued` matches `YYYY-MM`; `facets` is **rejected** (not yet implemented); `anchor_before_id` ∈ the derived entry-id set (from `deriveEntryIds()`, the same set that already validates `bullet_order` keys) — an unknown anchor or malformed file **fails fast** with a clear message before any render. **Shape-first gating:** before deciding whether to read `certifications.json`, validate `include_certs` shape and duplicates from the variant config alone. Only a valid non-empty string array or the literal `"all"` requests certs and triggers file load; an empty array `[]` requests nothing (file optional). Malformed values (`123`, object, mixed array, duplicate ids) fail as config errors even when the cert data file is absent. When no variant requests certs the file is never touched and may be absent (default/no-cert path unaffected); a requested-but-absent or malformed file **is** a fail-fast error (certs asked for, no cert data).
- The old frontmatter `render: true` flag is dropped — per-variant `include_certs` replaces a global on/off (this retires the "unused render flags" finding).

## Variant config — `include_certs`

New optional field on the existing JSON variant config:

```json
{
  "title": "Kyle Skrinak — Platform Operations Leader",
  "include_facets": ["platform-ops", "leadership", "cost"],
  "max_bullets_per_entry": 3,
  "include_certs": ["az-104", "aiops-foundation"],
  "anchor_before_id": "senior-it-systems-engineering-manager-digital-experience"
}
```

- `include_certs`: array of cert ids [chosen], or the literal `"all"`. Two-stage validation: (1) a **type/shape/duplicate** check (must be a string array or exactly `"all"`, no duplicate ids) that runs **before** cert-file loading and independent of `knownCertIds`, so malformed values (`123`, an object, a mixed array, repeated ids) are rejected as config errors even if `certifications.json` is absent; (2) an **id-membership** check against the injected `knownCertIds` set after cert data has been loaded. A bad id, duplicate id, or bad shape fails fast with a clear message *before any render*.
- `anchor_before_id` (optional): overrides `certifications.json`'s `anchor_before_id` for this variant only. Must be a member of the derived entry-id set (`KNOWN_ENTRY_IDS`). This lets a cert-emphasis variant place certs higher (e.g. before the first employer) without editing the durable cert data file. When omitted, falls back to `certifications.json`'s value.
- Omitted / empty ⇒ no certifications section (matches the default).

## Cert rendering (variant only)

- The builder reads `certifications.json`, resolves the requested ids (or `"all"`), and passes the resolved certs into the existing `page.evaluate` transform. **Single-arg bundling constraint:** Playwright's `page.evaluate(fn, arg)` accepts exactly **one** serializable argument — the current call passes `config` alone (`build-resume-variant.mjs:137,215`). All page-context data (config, resolved certs, expected count) must be bundled into a single payload object. A naive multi-arg call silently drops extra arguments. The page-context code must assert the payload shape (config exists; when certs are requested, the cert array exists and its length matches the expected count) before any DOM work — this catches dropped/botched payloads before the positional `<ul>` self-check (G‑4). **Order:** explicit `include_certs` renders in **requested order** (first id → first `<li>`); `"all"` renders in file order.
- **Anchor:** the transform locates the anchor `<h2>` via `content.querySelector('#' + CSS.escape(anchor_before_id))` (scoped to `.resume-content`) and injects an `<h2 class="cert-heading">Certifications</h2>` + `<ul>` — real DOM nodes built with `document.createElement`, not injected markdown — immediately before it. The effective `anchor_before_id` is the variant config's override if present, otherwise `certifications.json`'s value. The id is validated against the derived entry-id set pre-spawn; if the scoped DOM lookup still misses, the page-context transform throws and no PDF is written. Wrong placement is not warning-only.
- **Cert-heading styling (F‑1):** the injected `<h2>` **cannot** blindly reuse the resume's `h2` CSS. In `@media print` — the media the PDF renders — `resume.css` styles every entry `h2` as `display: inline` (`:178`) with `h2::after { content: ", " }` (`:192`), a rule built to run the role into the inline employer `<p>` that follows (`h2 + p { display: inline }`, `:197`). The cert `<h2>` is followed by a `<ul>`, not an employer `<p>`, so those inherited rules would print a dangling "**Certifications,**" (orphan comma, then the list breaks to the next line). Fix **without touching the shipped stylesheet**: the transform injects a scoped `<style>` node — `.resume-content h2.cert-heading { display: block } .resume-content h2.cert-heading::after { content: none }` — keeping `resume.css` byte-identical (it is a resume-PDF hash input **and** the published default's stylesheet) and the override contained to the variant tool. `::after` is a pseudo-element, so it **cannot** be reset via an inline `style` attribute on the node — a rule (class selector) is required, hence the injected `<style>`, not `element.style`. **The sibling `h2::before` block rule (`resume.css:181` — `content:""; display:block; margin-top:0.75rem`, with a `:first-of-type` reset at `:188`) is *deliberately not* overridden (G‑2):** with the cert heading now `display:block`, `::before` supplies the same inter-entry top gap every other entry gets, and the heading's own screen-rule `margin-top:0.75rem` collapses with it rather than doubling — so certs sit above the education block with correct spacing. Only `display` and `::after` need neutralizing; carrying the plan's "can't blindly reuse the entry-`h2` CSS" thesis through to `::before` confirms it is a feature, not a leak. The cert `<ul>`/`<li>` **do** reuse the existing list CSS cleanly (same `.resume-content ul`/`li` selectors, same depth).
- **Injection order:** the cert section is added **after** the existing bullet-walker loop, so the injected `<ul>` is **exempt** from facet-filtering, `max_bullets_per_entry`, and `bullet_order` — certs are all-or-nothing, never silently capped or reordered by a job-bullet knob. (Injecting before the loop would let `max_bullets_per_entry: 3` silently drop the 4th+ cert — explicitly avoided.)
- **List item:** each `<li>` renders the cert `name`, optionally followed by `— {issuer}` and `({issued})` when present. Only `name` is content-verified — and verified **positionally**, asserted inside the injected `<ul>` (G‑4), not merely present somewhere in the body.
- **One-page budget:** certs compete for space like any other content — the existing one-page gate (`countPdfPages`) fails the variant build if the additions overflow, and the fix is to trim **job** bullets (`max_bullets_per_entry` / `exclude_facets`) **or select fewer certs** (`include_certs` — certs are all-or-nothing, so dropping ids is the only cert-side lever), consistent with today's "trim weakest bullets first" rule. Each resolved cert `name` is passed as `requireText` so `verifyRenderedContent` asserts it in the rendered `innerText` before the PDF is written (see Scope → `resume-render.mjs`) — a whole-body cross-source check **secondary** to the positional `<ul>` self-check (G‑4), since a whole-body `includes` can false-pass on a name that appears elsewhere. **Message hint:** the one-page error message must conditionally mention certs as a possible overflow cause, but only on variant builds that include certs. **Null-safety constraint:** `renderResumePdf` destructures `expectedOverrides` with no default, and the CI path (`print-resume-pdf.mjs`) passes nothing — so `expectedOverrides` is `undefined` on the default path. All accesses to `expectedOverrides` in `renderResumePdf` (including the error message block) must use optional chaining or a default parameter. Note that `verifyRenderedContent` has its own default (`= {}`), so the two callsites have **different** null-safety — the same expression cannot be copied between them without checking.

## Accepted limitations (disposability-justified)

- **Positional bullet reorder** (`bullet_order`) stays index-based against the post-filter list — the known foot-gun is retained because variants are visually checked and disposable; not worth adding stable bullet ids to the markdown.
- **`data-facets` in public HTML** and **inline facet tags in the markdown** remain — cosmetic, SEO-inert, accepted.
- **Custom `unified()` processor** in `astro.config.ts` stays (drives `remark-facets`) — one low-churn version-pin watch-item; may be reverted later if `remark-facets` runs cleanly in the plain `remarkPlugins` array (out of scope here).
- **Slug-equivalence dependency** — the cert anchor resolves `anchor_before_id` against DOM ids from Astro's rehype-slug, while pre-validation derives the same ids with standalone `github-slugger`. `bullet_order` already relies on this equivalence; the cert anchor is now a **second** feature that breaks if the two slug implementations ever diverge. One low-probability watch-item, accepted — and **de-fanged** for the cert path: a divergence makes the scoped anchor lookup miss, which now fails hard before PDF write rather than appending in the wrong place.
- **`--base-url` + cert anchor interaction** — if `--base-url` points to a deployed site with stale content (e.g. a heading was renamed locally but not yet deployed), `anchor_before_id` passes pre-validation against the local source but the DOM lookup fails on the remote page. This is a pre-existing class of issue (variant filtering already depends on the remote DOM matching local source), but cert injection surfaces it more visibly because the anchor lookup fails hard rather than silently filtering. Accepted — the workaround is to not use `--base-url` with content that diverges from the local source.
- **Concurrent variant builds** — two simultaneous variant builds on the default port (4323) will collide. Override with `RESUME_PREVIEW_PORT` per invocation. Accepted for a single-user, single-invocation tool.

## Verification / control model

- **Default:** byte-identical render is the proof; `test:visual` stays green with no baseline change. No pixel-perfect migration gate is needed because the default does not change.
- **Variant:** the existing one-page gate + payload-shape assertion + hard anchor failure + **positional cert-injection self-check** (each `name` asserted inside the injected `<ul>`, G‑4) + content-presence verification (`requireText`, secondary) + a one-time visual eyeball. No pixel baseline for disposable output.

## Rollback

Revert the feature PR. No data migration required — the feature is purely additive (new file, new config field, new code paths). The default resume is byte-identical before and after, so revert produces no visible change to the published site.
